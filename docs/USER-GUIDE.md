# NoteSeed 产品使用说明书

> **Seed the web into your second brain.**
> 把每一次浏览，都变成一次沉淀。

---

## 这是什么？

你一定经历过这样的场景：

- 读到一篇绝佳的技术教程，收藏了，半年后再也找不到
- 看到一条行业新闻，想记下来，但手工复制粘贴太麻烦
- 打开收藏夹，200 条链接，一半已经 404，另一半忘了当初为什么存

**NoteSeed（种笔记）** 就是为了解决这个问题而生的。

它是一款 Chrome 浏览器扩展，做一件事：**把你正在看的网页，10 秒内变成一张结构化知识卡片，直接种进你已有的笔记系统**。

不是粗暴地"剪藏原文"，而是像一个聪明的助理——先判断你在看什么类型的内容，再按最适合的方式提炼精华，最后用你习惯的格式存到你常用的工具里。

---

## 产品理念

### 三条红线

1. **不做笔记系统的竞争者，做它们的入口。** 你的 Memos、飞书、Get 笔记是"主干"，NoteSeed 只是连通主干的"根须"。
2. **Skills 优先，模型其次。** 业务价值由 Skills 的组合决定，AI 模型只是执行器。换模型不影响产品行为。
3. **10 秒原则。** 从点击到卡片生成完成，中位耗时 ≤ 10 秒。任何功能若破坏这一节奏，延后做或不做。

### 和其他工具有什么不同？

| 对比维度 | Pocket / Instapaper | Notion Web Clipper | NoteSeed |
|----------|--------------------|--------------------|----------|
| 做什么 | 保存原文稍后读 | 裁剪网页到 Notion | 按类型智能提炼成卡片 |
| 输出物 | 原始全文 | 格式化网页副本 | 结构化 Markdown 卡片 |
| 内容策略 | 一刀切 | 一刀切 | **8 种页面类型 × 专属提炼策略** |
| 归宿 | 自有应用 | 绑定 Notion | Memos / 飞书 / 更多 |
| 长期价值 | 链接墓地 | 格式混乱的剪报 | 可检索、可复用的知识库 |

一句话总结：别的工具做"收藏"，NoteSeed 做"理解+提炼+分发"。

## 快速上手

### 第一步：安装

1. 打开 Chrome，进入扩展商店（或加载本地开发版本）
2. 搜索 **NoteSeed**，点击「添加至 Chrome」
3. 浏览器右上角会出现一颗小种子 🌱 图标

### 第二步：配置

点击扩展图标 → 进入「设置」页面：

1. **通用** Tab → 确认后端地址（本地默认 `http://localhost:3000`，云服务器填对应地址）
2. **AI 模型** Tab → 选择 AI 提供者（Anthropic / OpenAI / DeepSeek 等），填入 API Key 和模型名
3. **凭证** Tab → 连接你的笔记系统：
   - **Memos**：填入你的 Memos 服务地址和 Access Token → 测试连接 → 保存

> 所有凭证均通过 AES-256-GCM 加密存储，前端不存明文，可以放心使用。

### 第三步：开始制卡

1. 打开任意网页（比如一篇技术博客）
2. 点击浏览器工具栏的 🌱 图标 → 侧边栏展开
3. NoteSeed 自动识别页面类型（比如「教程」）
4. 点击 **⚡ 一键制卡**
5. 等待几秒，卡片生成完毕，可预览和编辑
6. 选择保存目标 → 点击 **🚀 保存**

就这么简单。

## 核心亮点：Skills 智能技能体系

Skills 是 NoteSeed 的灵魂。

和市面上所有剪藏工具"一个 prompt 打天下"的做法不同，NoteSeed 把"网页→卡片"这件事拆成了 **8 个独立的智能技能**，每个技能各司其职，像流水线一样协作。

### 为什么要这么做？

想象一下：一篇教程和一条新闻，需要提炼的东西完全不同。

- 教程你关心的是：**前置条件是什么？步骤是什么？有什么坑？**
- 新闻你关心的是：**谁做了什么？什么时候？有什么影响？**
- 观点文你关心的是：**核心论点是什么？论据有哪些？有没有反驳？**

用一个通用摘要来处理所有类型，就像用一把螺丝刀拧所有螺丝——能拧，但拧不好。

NoteSeed 的做法是：**先识别你在看什么，再决定怎么提炼。**

### Skills 全景

Skills 分为三层，对应知识从"采集"到"沉淀"的完整生命周期：

```
           🌱 种子层 — 感知与理解
     ┌──────────────┬──────────────┐
     │  PageSense   │Contextualizer│
     │  页面感知     │  上下文补全   │
     └──────┬───────┴──────┬───────┘
            │              │
           🌿 生长层 — 提炼与成型
     ┌──────┴───┬──────────┴───┬────────────┐
     │ Distiller│    Tagger    │ Cardwright  │
     │ 精华提炼  │   智能标签    │  卡片铸造   │
     └──────┬───┴──────┬───────┴──────┬──────┘
            │          │              │
           🌳 扎根层 — 分发与沉淀
     ┌──────┴──────────┴──────────────┴──┐
     │     Dispatcher     Chronicle      │
     │     目的地分发       历史沉淀       │
     └──────────────────────────────────┘
```

下面逐一介绍每个 Skill 的能力和亮点。

---

### 🌱 Skill 1：PageSense — 页面感知

> "判断这是一粒什么种子。"

当你打开侧边栏的那一刻，PageSense 已经开始工作了。它会分析当前页面的 URL、DOM 结构、文本特征，判断这是一个什么类型的页面。

#### 支持的 8 种页面类型

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **tutorial** | 教程 / 指南 | MDN 文档、技术博客教程、"手把手教你"系列 |
| **opinion** | 观点 / 评论 | 专栏文章、行业分析、个人博客观点 |
| **news** | 新闻 / 资讯 | 科技新闻、行业快讯、产品发布 |
| **doc** | 文档 / API | GitHub README、官方 API 文档、技术规范 |
| **tool** | 工具 / 产品 | SaaS 产品首页、Chrome 扩展页、开源项目 |
| **resource** | 资源列表 | Awesome 列表、工具合集、学习路线图 |
| **longform** | 长篇深度 | 万字研报、深度调查、长篇叙事 |
| **discussion** | 讨论 / 问答 | Hacker News、Reddit、知乎问答、Stack Overflow |

#### 聪明在哪？

PageSense 不是简单地把页面内容丢给 AI 让它猜。它采用**多信号融合**的方式：

1. **域名特征**：`github.com/*/README.md` → 直接判定为文档，`zhihu.com/question/*` → 直接判定为讨论
2. **DOM 结构**：`<code>` 代码块多 → 偏教程/文档，`<blockquote>` 引用多 → 偏观点
3. **文本信号**：出现"Step 1/2/3" → 教程，出现"I think"、"我认为" → 观点
4. **AI 仲裁**：当以上信号不够确定时，才调用 Claude Haiku 做最终判断

**信号足够强（置信度 ≥ 80%）时，直接返回结果，根本不调 AI**，又快又省。

当识别结果不确定（置信度 < 60%）时，界面会展示类似"教程（65%），可能是文档（30%）"的提示，你可以一键切换。

### 🌱 Skill 2：Contextualizer — 上下文补全

> "为种子补齐水土信息。"

一张知识卡片半年后再看，最怕的就是"忘了这是从哪来的、谁写的、什么时候写的"。Contextualizer 在卡片生成阶段就自动帮你补全这些信息。

#### 补全哪些信息？

| 字段 | 来源 | 说明 |
|------|------|------|
| **作者** | OG meta / JSON-LD / 正文推断 | 多源交叉验证 |
| **发布日期** | meta 标签 / 正文日期 / URL 日期 | ISO 格式标准化 |
| **语言** | Unicode 字符范围检测 | 中文 / 英文 / 日文等 |
| **阅读时间** | 中文 300 字/分，英文 200 词/分 | 本地计算，不调 AI |

#### 聪明在哪？

- **先本地后 AI**：阅读时间、语言检测完全在本地完成。只有在页面 meta 信息缺失时才调用 AI 补全作者和日期，节省 API 调用
- **"脱离网页可读"检测**：如果内容中出现"本文"、"上一节"、"如前所述"等依赖上下文的表述，会自动替换成具体指代，让卡片离开原文也能独立阅读
- **系列文识别**：如果这篇是系列文章的其中一篇（如"React 18 深入系列 · 第 3 篇"），会自动识别系列标题

### 🌿 Skill 3：Distiller — 精华提炼

> "把一整片网页熬成一小勺精华。"

**这是 NoteSeed 最核心的 Skill，也是和所有竞品拉开差距的地方。**

Distiller 不做"通用摘要"，而是根据 PageSense 识别出的页面类型，调用**完全不同的提炼策略**，产出不同的结构化字段。

#### 不同类型，不同提炼

| 页面类型 | 提炼出什么 | 为什么这么提炼 |
|----------|-----------|---------------|
| **tutorial** | 摘要 + 前置条件 + 步骤列表 + 注意事项 | 教程的价值在步骤，不在废话 |
| **opinion** | 摘要 + 核心论点 + 论据 + 金句 + 反驳观点 | 观点文要区分"他说了什么"和"依据是什么" |
| **news** | 摘要 + 5W1H（谁/什么/何时/何地/影响） + 关键事实 | 新闻要提炼事实，不是情绪 |
| **doc** | 摘要 + 用途 + API 接口/参数 + 代码示例 | 文档的价值在可操作的细节 |
| **tool** | 摘要 + 使用场景 + 优劣势 + 定价 | 工具评估需要决策信息 |
| **resource** | 摘要 + 描述 + 亮点 + 适合谁 | 资源列表要快速判断值不值得看 |
| **longform** | 摘要 + 文章大纲 + 核心洞察 | 长文先看骨架再决定要不要深读 |
| **discussion** | 原始问题 + 高赞回答 + 共识 + 争议点 | 讨论的价值在多元观点 |

#### 聪明在哪？

- **类型专属 Prompt**：每种页面类型背后是一套精心打磨的 Prompt 模板，不是丢一个"请帮我总结"了事
- **事实与观点分离**：对于新闻和观点文，Distiller 会区分"事实陈述"和"作者观点"，分别标注，避免把别人的观点误当事实
- **使用 Claude Sonnet 模型**：这是管线中唯一使用高能力模型的环节——因为内容提炼的质量直接决定卡片价值，值得用最好的模型

#### 举个例子

假设你在看一篇《React useMemo 性能优化指南》（tutorial 类型），Distiller 会提炼出：

```markdown
## 摘要
本文介绍了 React useMemo 的三种典型使用场景，
帮助开发者避免不必要的重渲染。

## 前置条件
- 熟悉 React Hooks 基础
- 了解 React 渲染机制

## 步骤
1. 识别性能瓶颈：使用 React DevTools Profiler
2. 用 useMemo 缓存昂贵计算
3. 用 React.memo 包裹纯展示组件
4. 用 useCallback 稳定回调引用

## 注意事项
- useMemo 不是万能药，小计算不需要缓存
- 依赖数组写错会导致缓存失效
```

而如果你在看一条《OpenAI 发布 GPT-5》的新闻（news 类型），Distiller 会提炼出：

```markdown
## 摘要
OpenAI 正式发布 GPT-5 模型，推理能力提升 40%...

## 5W1H
- **谁**：OpenAI
- **什么**：发布 GPT-5
- **何时**：2026 年 3 月 15 日
- **影响**：推理性能提升 40%，API 价格下降 30%

## 关键事实
- GPT-5 上下文窗口扩展至 256K
- 新增原生多模态理解
- 企业版同步上线
```

**同一个工具，面对不同内容，产出完全不同的知识结构。** 这就是 NoteSeed 的核心竞争力。

### 🌿 Skill 4：Tagger — 智能标签

> "让这张卡未来能被检索到。"

一张卡片如果搜不到，和没存一样。Tagger 自动为每张卡片推荐 3~5 个标签。

#### 不只是"生成标签"

| 能力 | 说明 |
|------|------|
| **标签推荐** | 基于卡片内容生成 3~5 个标签 |
| **分类** | 宽泛分类，如"技术"、"商业"、"设计" |
| **主题** | 具体主题，如"React Hooks"、"容器化" |

#### 聪明在哪？

- **向你的既有标签靠拢**：Tagger 会读取你笔记系统中的历史标签，优先复用已有标签，避免 `#React`、`#react`、`#reactjs` 三个标签并存的混乱
- **层级标签**：支持 `#技术/前端/React` 这种层级结构，方便后续做树形组织
- **新颖度评分**：当一张卡的内容和你之前存过的高度重合时，会提示"你 2 周前存过类似内容"，避免重复囤积

---

### 🌿 Skill 5：Cardwright — 卡片铸造

> "把零件拼成一张端正的卡。"

Cardwright 是管线的最后一个生成环节，负责把前面所有 Skill 的输出组装成最终的 Markdown 卡片。

#### 6 种卡片模板

| 模板 | 说明 |
|------|------|
| **平衡** | 默认模板，摘要 + 要点 + 引述 |
| **精简** | 极简输出，一句话摘要 + 3 个核心要点 |
| **详细** | 完整卡片，含引述、不同意见与反论点 |
| **教程提炼** | 提取步骤、前置条件、代码片段 |
| **观点摘要** | 聚焦论点、论据、立场分析 |
| **自定义提示词** | 输入任意提示词，AI 按你的要求生成 |

#### 特点

- 优雅处理缺失字段（某个字段提取失败不会导致卡片崩溃）
- 保存到 **Memos** → 自动格式化为 Markdown + `#标签`
- **不调 AI**：纯 TypeScript 模板引擎，速度极快，零 API 成本

---

### 🌳 Skill 6：Dispatcher — 目的地分发

> "把卡种到对的土里。"

Dispatcher 是 NoteSeed 从"AI 演示品"升级为"工作流工具"的关键。

#### 支持的保存目标

| 目标 | 模式 | 特殊处理 |
|------|------|----------|
| **Memos** | Compact（1500 字内） | 标签拼到正文末尾，默认 private |
| **飞书文档** | Full Markdown | 自动建立在指定文件夹，支持富文本 |
| **更多平台** | 开发中 | Get 笔记、金山文档等 |

#### 聪明在哪？

- **同时分发**：勾选多个目标，Dispatcher 并发执行，部分失败不影响整体
- **失败不丢数据**：如果目标系统临时不可达，卡片会自动暂存到本地（IndexedDB），下次上线时自动补发
- **目的地推荐**：Dispatcher 会根据页面类型推荐默认目的地（教程默认 Memos、长文默认飞书），你选一次就记住

---

### 🌳 Skill 7：Chronicle — 历史沉淀

> "记住每一颗种过的种子。"

Chronicle 管理你的本地制卡历史，让 NoteSeed 成为一个"有记忆的助手"。

#### 能力

| 功能 | 说明 |
|------|------|
| **本地历史** | 最近 100 张卡片存于浏览器本地（IndexedDB），不依赖云端 |
| **去重提醒** | 基于 URL + 内容指纹，重复保存会提示"你 3 天前存过" |
| **系列聚合** | 如果多张卡属于同一系列（如"React 18 深入"），自动聚合 |

### Skills 完整流程

当你点击「一键制卡」后，幕后发生了什么：

```
你点击 ⚡ 一键制卡
       │
       ▼
  ① PageSense 分析页面 ─── "这是一篇教程，置信度 92%"
       │
       ▼
  ② Contextualizer 补全 ── "作者：张三，发布于 2025-09-12，阅读 8 分钟"
       │
       ▼
  ③ Distiller 按类型提炼 ── 摘要 + 步骤 + 前置条件 + 注意事项
       │
       ▼
  ④ Tagger 推荐标签 ────── #前端 #React #性能优化
       │
       ▼
  ⑤ Cardwright 渲染卡片 ── 组装成最终 Markdown
       │
       ▼
  📋 卡片预览（你可以编辑）
       │
  你点击 🚀 保存
       │
       ▼
  ⑥ Dispatcher 分发 ────── 并发写入 Memos + 飞书
       │
       ▼
  ⑦ Chronicle 记录 ─────── 本地存档 + 去重索引
       │
       ▼
  ✅ 完成！
```

**全程 ≤ 10 秒。** 你能看到每一步的进度提示。任何一步失败，后续步骤会降级到默认值继续执行，不会让你空手而归。

## 设置说明

点击侧边栏右上角 ⚙ 图标，进入设置页面。

### 通用设置

- **后端地址**：本地开发默认 `http://localhost:3000`，云部署填服务器地址

### AI 模型配置

- 支持 Anthropic、OpenAI 及兼容协议（DeepSeek / Moonshot / GLM / Ollama 等）
- AI Key 存储在扩展本地，不上传服务器

### 笔记系统连接

| 系统 | 需要什么 | 在哪找 |
|------|----------|--------|
| **Memos** | 服务地址 + Access Token | Memos 设置 → API → 创建 Access Token |
| **飞书** | App ID + App Secret | [飞书开放平台](https://open.feishu.cn/) → 创建应用 |

配置完成后可点击「测试连接」验证是否正常。

### 偏好设置

| 设置项 | 说明 |
|--------|------|
| **默认模板** | 设为 `auto` 让 PageSense 自动选择，或指定固定模板 |
| **默认保存目标** | 设定默认勾选的保存目标 |
| **Skills 开关** | Tagger 等可选 Skill 可独立关闭 |

## 常见问题

### "页面内容为空"怎么办？

部分 SPA 页面（如 React/Vue 单页应用）内容由 JavaScript 动态渲染。等页面加载完毕后，点击「重新识别」即可。

### 页面类型识别不准确？

直接在顶部点击类型标签手动切换。切换后会用新类型重新走 Distiller 提炼管线。

### 制卡太慢（超过 20 秒）？

通常瓶颈在 Distiller（Sonnet 模型）步骤。可能原因：
- 页面正文过长 — 尝试选中关键段落后制卡（选区模式）
- 网络延迟 — 检查到 Anthropic API 的网络连通性
- API 限流 — 检查 Anthropic Console 用量

### Memos 保存失败（401）？

Memos Access Token 过期。在设置页面更新 Token，点击「测试连接」验证。

### 飞书保存失败？

常见原因：
1. App ID 或 App Secret 填写错误
2. 飞书应用权限不足（需要 `docx:document:create` 等权限）
3. 应用未发布到可用范围

### 能离线使用吗？

NoteSeed 使用 IndexedDB 做本地暂存。如果网络不可用，卡片会缓存在本地，网络恢复后自动补发。但生成卡片本身需要调用 AI 服务，离线时无法生成新卡片。

## 隐私与安全

| 项目 | 处理方式 |
|------|----------|
| **页面内容** | 仅用于当次生成，不在服务端留存 |
| **API 凭证** | AES-256-GCM 加密存储，前端不存明文 |
| **本地数据** | 历史卡片、风格档案、标签缓存均存于浏览器本地 |
| **浏览记录** | NoteSeed 仅在你主动点击时采集当前页，不收集浏览历史 |
| **权限** | 仅申请 `activeTab`、`sidePanel`、`storage`、`scripting` 四项必要权限 |
