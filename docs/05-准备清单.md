## 第一部分：开工前准备清单

清单分五类。**带 🔴 标记的是硬性前置**，缺一项就开不了工；**🟡 是强烈建议**，不准备也能起步但中途一定会回来补；**🟢 是锦上添花**。

---

### A. 账号与密钥类（最容易卡住，请优先搞定）

#### 🔴 A1. Anthropic Claude API 密钥

**为什么要**：NoteSeed 的所有 Skills（PageSense / Distiller / Tagger 等）都依赖 Claude API，这是**最核心的依赖**。没有它，Phase 3 完全无法开始。

**怎么获取**：
1. 访问 https://console.anthropic.com/
2. 注册账号（建议用工作邮箱）
3. 进入 `Settings → API Keys`，创建新 key
4. **重要**：绑定信用卡并充入至少 **$20** 启动额度（开发期足够用）

**保存到哪**：
- 暂时存在密码管理器（1Password / Bitwarden）
- 稍后写入 `.env` 文件的 `ANTHROPIC_API_KEY` 字段

**预计耗时**：15 分钟
**预计费用**：$20 启动金，MVP 开发完成预计消耗 $10-30

---

#### 🟡 A2. OpenAI API 密钥（备用）

**为什么要**：MVP 可不用，但 Phase 3 的容灾备份依赖它。也用于生成 Embedding（Tagger 去重用）。

**怎么获取**：
1. 访问 https://platform.openai.com/
2. 注册 + 绑卡 + 充值 $10
3. 创建 API Key

**可以推迟**：如果预算紧张，这一步可以推到 v1.1 再做。MVP 用 Claude 单一来源即可。

**预计耗时**：10 分钟

---

#### 🔴 A3. Memos 测试实例

**为什么要**：MVP 要跑通"存到 Memos"这条路径，你得有一个可以写入的 Memos 实例做测试。

**两个选项**：

**选项 1（推荐）：本地 Docker 跑一个**
```bash
docker run -d \
  --name memos \
  -p 5230:5230 \
  -v ~/.memos:/var/opt/memos \
  neosmemo/memos:stable
```
- 访问 http://localhost:5230 完成初始化
- 在 `Settings → My Account` 生成 Access Token

**选项 2：用你已有的自建实例**
- 如果你本来就在用 Memos，直接拿一个测试 token 就行

**需要准备好什么**：
- Memos 实例的 **baseUrl**（如 `http://localhost:5230`）
- 一个**有写入权限的 Access Token**

**预计耗时**：30 分钟

---

#### 🔴 A4. 飞书开放平台应用

**为什么要**：MVP 要跑通"存到飞书"，需要飞书应用凭证。

**怎么做**：
1. 访问 https://open.feishu.cn/
2. 注册开发者账号（需要飞书账号）
3. 创建一个**企业自建应用**
4. 获取：
   - `App ID`
   - `App Secret`
5. 在"权限管理"中申请以下权限：
   - `docx:document` — 创建云文档
   - `docx:document:readonly` — 读取云文档
   - `drive:drive` — 访问云空间
6. 创建测试用的**云空间文件夹**，记下 `folder_token`

**⚠️ 注意**：
- 飞书应用从创建到权限生效可能要 **1-2 小时**（审批流程）
- 建议**今天就申请**，不要等到 Phase 4 开始才申请

**预计耗时**：1 小时操作 + 1-2 小时审批

---

#### 🟢 A5. GitHub 账号与仓库

**为什么要**：代码托管、CI/CD、issue 追踪。

**怎么做**：
1. 创建私有仓库 `noteseed`（或 public 随你）
2. 建议初始化时**不要**勾选 README/LICENSE/gitignore（AI 会在 Phase 0 生成）
3. 生成一个 Personal Access Token（给本地 push 用）

**预计耗时**：5 分钟

---

#### 🟢 A6. 域名（可延后到上线前）

**为什么要**：后端 API 需要一个域名（如 `api.noteseed.app`）。

**建议**：MVP 开发期用 `localhost` 即可，上线前 2 周再买域名。

**推荐注册商**：Cloudflare Domains（配套 CDN + DNS 省事）

### C. 项目资产类

#### 🔴 C1. 产品图标

**为什么要**：插件必须有图标才能加载到 Chrome，Phase 5 就会卡住。

**需要准备**：
- **16 × 16** 像素（工具栏图标）
- **48 × 48** 像素（扩展管理页图标）

**三种准备方式**：

**方式 1：自己设计**（如果你有设计能力）

- 主色 `#4A8B5C`（PRD 定的种子绿）

**方式 2：外包**
- 闲鱼 / Fiverr 找人做，¥100-500，3 天内出图

**方式 3：AI 生成**
- 用 Midjourney / DALL-E 生成，再用 Figma 导出三个尺寸
- Prompt 参考：`minimalist seed sprouting icon, flat design, green #4A8B5C, white background, app icon style`

**预计耗时**：1 天（含生成和定稿）

### D. 文档类（这是最容易被忽略但最关键的一项）

#### 🔴 D1. 把前面四份文档集中放在一起

**为什么要**：AI 编码代理工作时会反复读取这些文档。它们必须**在项目能找到的地方**。

**建议做法**：

在项目根目录创建 `docs/` 文件夹，放入：

```
noteseed/
├── docs/
│   ├── 01-PRD.md                    # 产品需求文档
│   ├── 02-技术附录.md                # 技术架构与 API
│   ├── 03-开发任务书.md              # Phase 分解（AI 主要读这份）
│   └── 04-全解读.md                  # 可选，AI 不一定读
```

**关键**：开发任务书（03）必须在项目里。AI 执行时会频繁引用 Phase / Task 编号。

---

#### 🔴 D2. 创建项目规则文件（Project Rules）

**这是最重要的一步**。它决定 AI 每次回答前都记住什么。

**如果用 Claude Code**，创建 `CLAUDE.md`：
**如果用 Cursor**，创建 `.cursorrules`：

内容建议（可直接复制使用）：

```markdown
# NoteSeed 项目规则

你正在开发 NoteSeed——一款浏览器插件，把网页一键变成结构化 Markdown 知识卡片。

## 必读文档
开始任何任务前，必读：
1. `docs/03-开发任务书.md` — 找到当前 Phase 和 Task
2. `docs/01-PRD.md` — 查询功能需求细节
3. `docs/02-技术附录.md` — 查询技术选型和 API

## 核心约束（不可违反）
- 技术栈固定：React 18 + TypeScript 5 + Vite + CRXJS + Fastify + Prisma + PostgreSQL
- 包管理器只用 pnpm，禁用 npm / yarn
- TypeScript 严格模式，禁用 any
- 所有对外 API 用 Zod 校验
- 所有日期字段用 ISO 8601 字符串
- 每个 Skill 独立目录、独立文件
- Git commit 格式：`[P{phase}-T{task}] <description>`

## Phase 工作流
- 按顺序执行 Phase 0 → 8
- 每完成一个 Phase，产出 `docs/progress/phase-{N}-DONE.md` 进度报告
- 未完成当前 Phase 不允许开下一 Phase
- 遇到决策模糊，优先查任务书的「默认决策表」

## 汇报要求
- 每完成一个 Task，简要说明：改动了哪些文件、下一步做什么
- 遇到不确定的技术决策，停下来问我，不要自行发挥
- 遇到需要我申请密钥/权限的情况，停下来告诉我

## 禁止事项
- 禁止跳过测试
- 禁止自行修改 PRD 定义的需求
- 禁止提交 .env 或任何密钥到 Git
- 禁止安装未在技术附录中列出的依赖（如必须，先问我）
```

**创建时机**：Phase 0 之前就放好，AI 第一次工作就读到。

**预计耗时**：30 分钟（复制 + 根据实际调整）

---

#### 🟡 D3. `.env.example` 清单心里有数

在 Phase 0，AI 会生成 `.env.example`。你需要提前**心里清楚**需要哪些环境变量：

```env
# 数据库
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/noteseed
REDIS_URL=redis://localhost:6379

# JWT
JWT_SECRET=<32 字节随机字符串>
JWT_ACCESS_EXPIRES=1h
JWT_REFRESH_EXPIRES=30d

# 加密
CREDENTIAL_ENCRYPTION_KEY=<32 字节 base64 字符串>

# AI 服务
ANTHROPIC_API_KEY=<你的 Claude 密钥>
OPENAI_API_KEY=<你的 OpenAI 密钥，可选>

# 飞书（Phase 4 用到）
FEISHU_APP_ID=<你的飞书 App ID>
FEISHU_APP_SECRET=<你的飞书 App Secret>

# 运行环境
NODE_ENV=development
PORT=3000
LOG_LEVEL=info
```

**生成随机字符串**：
```bash
# JWT_SECRET
openssl rand -base64 32

# CREDENTIAL_ENCRYPTION_KEY
openssl rand -base64 32
```

提前把这些生成好存在密码管理器里，AI 创建 `.env` 时直接填。

### E. 测试资源类

#### 🔴 E1. 测试网页 fixture 集合

**为什么要**：Phase 3 的 Skills 需要在真实网页上测试。你得提前准备好**8~10 个代表性网页的 HTML 快照**，AI 用它们做单元测试。

**怎么准备**：

找 8~10 个典型网页，每个类别至少 1 个：

| 类型 | 推荐示例 |
|---|---|
| tutorial | 掘金 / MDN / 阮一峰博客的教程文 |
| opinion | Medium / 少数派 / 虎嗅的观点文 |
| news | 36kr / TechCrunch 的新闻 |
| doc | GitHub README / API 文档 |
| tool | Product Hunt 产品页 |
| resource | Awesome List 仓库 |
| longform | Paul Graham 的 essay |
| discussion | HackerNews / 知乎问题 |

**保存方式**：
- 浏览器保存完整网页 HTML（Ctrl+S → Webpage Complete）
- 放到 `packages/skills/__fixtures__/pages/` 目录
- 按类型命名：`tutorial-01-docker.html`、`opinion-01-ai-education.html`

**预计耗时**：1 小时（找 + 保存）

**更高级的做法**：对每个 fixture，你人工整理出一份"理想的卡片输出"作为基准。

这样 AI 在开发 Skills 时可以**对照评分**。

**建议做**：至少给 tutorial 和 opinion 各准备 2 份"理想卡片"，AI 会照着这个质量去调 prompt。

## 第二部分：你在开发过程中扮演的角色

AI 负责写代码，**但一个 AI 编码项目从来不是"你丢需求 AI 干活"那么简单**。你全程扮演五个角色：

### 角色 1：产品负责人（Product Owner）

**你的核心职责**：确保开发方向和产品意图一致。

**具体工作**：
- 开发过程中遇到 **PRD 没覆盖到的功能细节**，由你拍板
- 看到 AI 产出的界面和交互不符合预期时，**给出具体反馈**（不是"不对"而是"应该是 X 而不是 Y"）
- 权衡"做不做这个功能"的取舍

**频率**：几乎每天都有

**典型场景**：
> AI：PageSense 识别出页面类型后，用户点击"重新识别"时应该清空当前卡片还是保留？
>
> 你：保留。用户可能只是想换个模板，不想重新生成。

---

### 角色 2：决策裁判（Decision Maker）

**你的核心职责**：在 AI 停下来问你的时候，快速决策。

**具体工作**：
- **技术选型决策**：AI 说"这里用 A 还是 B？"——你给答案
- **需求冲突决策**：两个需求打架时选哪个
- **成本/时间决策**：某个功能花太长时间，是继续做还是砍掉

**频率**：每周 3-5 次

**典型场景**：
> AI：Distiller 在处理超长网页（>20000 字）时，是截断还是分段处理？截断省成本但丢信息，分段完整但耗时翻倍。
>
> 你：MVP 先截断。v1.1 再做分段。

---

### 角色 3：质量审核员（QA Reviewer）

**你的核心职责**：每个 Phase 结束时做代码 review 和功能验收。

**具体工作**：
- 读 AI 产出的 `phase-{N}-DONE.md` 报告
- 逐条检查「验收清单」是否真的过了
- 手动测试一下关键功能（AI 说通了的你自己也点一遍）
- 发现问题 → 生成具体 issue → 让 AI 修

**频率**：每个 Phase 结束一次，MVP 共 9 次

**为什么你不能跳过**：
AI 会**倾向于说"通过了"**，即使某个细节没做好。你是最后一道防线。

---

### 角色 4：外部联络员（External Liaison）

**你的核心职责**：所有和"项目外世界"打交道的事。

**具体工作**：
- 申请 API 密钥
- 注册第三方平台账号（飞书开放平台等）
- 和域名注册商、云服务商打交道
- 应对 Chrome Web Store 审核
- 和可能的第一批用户沟通

**频率**：前期密集，后期零散

**为什么 AI 不能替代**：
这些事需要**真实身份 / 信用卡 / 法律主体**。AI 做不了。

---

### 角色 5：最终拍板人（Tiebreaker）

**你的核心职责**：AI 陷入两难时，你给最终答案。

**具体工作**：
- AI 反复在两个方案间徘徊时，你指定一个
- 出现没有"正确答案"的问题时（如"按钮放左边还是右边"），你决定
- 时间压力下砍功能时，你决定砍哪个

**频率**：每周 1-2 次

**心态**：**不要纠结。50 分的决定大于 90 分的不决定**。AI 卡住比做错更浪费时间。

---

## 第三部分：哪些事必须你亲自做（AI 不能替代）

下面这些事，**AI 不可能替你做，或者替你做了有风险**。务必亲自做。

### 必做 1：所有涉及真实身份的事

- 注册 Anthropic / OpenAI / 飞书账号
- 绑定信用卡、充值
- 签署开发者协议
- 申请 Chrome Web Store 发布者账号（需要 $5 年费 + 实名验证）

**⚠️ 重要**：**永远不要把你的信用卡信息、身份证号告诉 AI**，哪怕它"保证帮你自动化处理"。

---

### 必做 2：密钥的最终保管

AI 可以在代码里使用密钥，但**密钥本身的生成和存储在密码管理器里，只有你操作**。

**永远不要**：
- 让 AI "帮你保存密钥到某个文件"
- 把密钥直接贴在聊天里（即使是 Claude 这种安全的 AI 也要警惕，因为日志和历史可能泄漏）
- 把密钥 commit 到 Git（这是最常见的安全事故）

---

### 必做 3：决定"产品灵魂"的事

有些事不是技术问题，是**价值判断**：

- StyleMatcher 默认开启还是关闭？（隐私 vs 体验的权衡）
- 免费用户能用几张卡？（增长 vs 成本的权衡）
- 界面是极简还是信息密集？（新用户 vs 老用户的权衡）

这些问题 AI 可以给你**列选项和利弊**，但**最终决定必须你做**。因为这些决定合起来定义了你的产品**是什么样的产品**。

---

### 必做 4：真实用户调研

- 找 5-10 个潜在用户，**你亲自和他们聊**
- 看他们怎么用、什么时候困惑、什么时候兴奋
- AI 可以帮你整理访谈笔记，但**不能替你做访谈**

**这是最容易被忽略但最重要的事**。MVP 阶段最大的风险不是"代码写不出来"，而是"做出来没人用"。亲自接触用户是唯一解法。

---

### 必做 5：法律与合规决策

- 隐私政策的条款（可让 AI 起草，但必须你通读并对每一条负责）
- 数据处理协议
- 跨境数据传输的合规性（如果要做海外版）
